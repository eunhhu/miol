각 도메인별 개념은 격리된 생태계 내에서 최고성능을 위해 개발되어옴
그니까 db의 sql은 단순 데이터 저장의 효율을 위해 설계됐고 구조조차 단순 "데이터 저장"이라는 목적으로 디자인됨. 그렇기 때문에 table당 계층 상속이 불가하여 orm으로 모델간 relation을 걸어서 현대 사용되고 있음. 그니까 sql이 범용적으로 사용하기 위해 설계되었다는 거임. 얘가 웹에서 사용되든 게임에서 사용되든 그런건 신경안쓰고 그냥 sql방식대로 저장만 하면됨.
일단 이게 가장 크나큰 문제임. 호환을 위해 성능을 포기함.
기존 http 프로토콜도 마찬가지. 내가 랜딩웹앱을 만들든 게임웹앱을 만들든 같은 통신을 사용해서 오버헤드가 발생하기도 함.
그나마 요즘은 도메인 특화 툴, 라이브러리, 프레임워크 등등 개발 생태계가 바뀌어 왔지만 이것도 "도메인 특화"일 뿐이지, 결국 개발자가 가지고 있는 목표에 맞는 라이브러리나 프레임워크에 대한 리서치는 반강제임.
이 부분에서 갑자기 문득 떠오름. "프로젝트 특화" 프레임워크가 있다면 어떨까?

내가 A라는 기능을 위해 개발을 하게된다면 최종적인 아웃풋은 그 기능을 위해 최적화된 번들이 나오는거임.
이런 고효율 최적화를 위해서는 어느정도 호환성을 포기하고 이용자를 본인 생태계로 끌어들여야함.
하지만 그로인해 발생하는 생산성과 가치는 무시못함. 이는 애플 생태계가 증명해냄.

다른 도메인과의 관계에 따른 최적화를 통해 보다 효율적으로 기능하도록 하기 위해 생각하게됨.

마치 수술을 위해 여러 도구를 전부 좋은걸로 준비하는 것 보단, 의사가 해당 환자를 수술하는데 딱 필요한 도구만 최적의 루트로 바로바로 연계해서 수술할 수 있도록 준비하는 것. 심지어 물건을 바꿔줄 조수도 필요가 없음. 그냥 수술 전에 수술실 세팅과 루트, 환경이 해당 수술을 위해서 조성된 것처럼 꾸려짐.
