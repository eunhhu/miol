# orv Operational Surfaces

이 문서는 현재 CLI, editor/LSP/DAP, build/deploy, DB 운영 surface를 추적한다. 핵심 크레이트 구조와 데이터 흐름은 [ARCHITECTURE.md](ARCHITECTURE.md)에 둔다.

## CLI Surface

현재 `orv-cli`는 프로젝트 scaffold, 로드/해석/분석/실행, graph/origin 출력, editor/LSP/DAP bootstrap, build artifact 생성/검증/실행, lockfile 생성/검증, DB snapshot/migration workflow를 오케스트레이션한다.

주요 command:

- `orv init <dir> --name <name> [--template basic|shop]`
- `orv run/dev/check/dump/origins/graph/test`
- `orv editor snapshot/reveal/runtime/debug/run-debug/export/trace/trace-stream`
- `orv lsp snapshot/reveal/serve --stdio`
- `orv dap serve --stdio`
- `orv build <file-or-orv.toml> --out <dir> [--prod]`
- `orv add/remove`
- `orv lock [dir-or-orv.toml] [--check]`
- `orv fetch [dir-or-orv.toml] [--out <dir>]`
- `orv workspace new <member> [--root <dir>] [--name <name>]`
- `orv workspace graph [root] [--view] [--out <dir>]`
- `orv workspace lock [root] [--out <dir>]`
- `orv workspace fetch [root] [--out <dir>]`
- `orv workspace build [root] [--out <dir>] [--prod] [--incremental]`
- `orv verify-build/verify-artifact/check-artifact/check-build`
- `orv run-artifact/run-build/reveal`
- `orv db plan/verify/apply/migrate/rollback/backup/restore/recover/archive/squash`

Source-entry commands accept a single `.orv` file, an `orv.toml` with `[project].entry`, or a project directory containing `orv.toml`.

Reference HTTP routes now support named-schema request bindings: `@body: T`, `@query: T`, and `@form: T` run the same runtime validators as `T.parse(...)`, expose normalized request-state values to the remaining handler body, and short-circuit invalid input with a 400 `{ error: "validation_failed", fields: [...] }` response.

`orv init <dir> --template shop` writes the reference shop starter with `GET /` browser checkout forms, typed `@body: T` schema bindings on browser mutation routes, a customer-facing browser `GET /catalog` storefront catalog, browser `GET /cart` cart view, browser `GET /account/sessions` active session view protected by `@session required`, signup/login forms that store Argon2 `passwordHash` values with `hash.password` and verify them with `hash.verify`, a seeded reference admin member for local smoke sessions, `@Auth required role="admin"` protected admin dashboard/read models for catalog/order/payment/shipment/webhook/audit data and persistence paths, an env-overridable `SHOP_DATABASE_URL` DB adapter that defaults to SQLite-backed `data/shop.sqlite`, env-overridable payment/shipping adapters that default to file-backed commerce record logs under `data/`, reference session/role cookies for successful member login, reference CSRF hidden-token wiring for browser mutation forms, reference rate limits for login/checkout/webhook routes, a reference `POST /webhooks/stripe` signature verification and duplicate event endpoint, and generated build/deploy/native launcher guidance.

Build, server runtime, deploy, and native plan artifacts expose those reference security requirements as shared `runtime_features`: `auth_roles`, `session_cookies`, `csrf_protection`, and `rate_limit`. Route descriptors also carry per-route `policies[]` entries for source-backed `auth`, `session`, `csrf`, and explicit `rateLimit` domains plus built-in `rate_limit` defaults, and `orv reveal` includes those policies in production route payloads. Route handlers can override defaults with `@rateLimit key=@body.memberId limit=2 window="1m"` or disable a built-in hotspot default with `@rateLimit exempt`; the reference runtime applies those checks before handler execution. `orv verify-build` checks runtime feature equality and source origin containment for those route policies, so drift in the advertised security surface fails before deploy smoke or native launcher metadata is trusted.

`orv graph <file> --view --out <dir>` writes `graph.json` and a static `index.html` ProjectGraph view with source/semantic depth stats, node/edge visualization, node search/kind filtering, and origin rows.

`orv dev --hmr` writes `dev/session.json`, `dev/transport.json`, and `dev/hmr-client.js` for the reference EventSource HMR browser transport; `orv dev --watch` writes `dev/watch.json`; `orv dev --watch-loop [--watch-iterations <n>]` runs the poll-loop build/verify/run path while writing `dev/events.json`; and `orv dev --hmr --serve [--serve-port <port>]` starts the reference HTTP/1 HMR endpoint with `dev/server.json`, `/__orv/hmr/session`, and `/__orv/hmr/events`.

`orv lock [dir-or-orv.toml]` reads `[project]`, `[dependencies]`, and `[dev-dependencies]` from `orv.toml`, writes deterministic JSON `orv.lock` entries sorted by package name, preserves registry/path sources, preserves optional `auth_token_env` names without storing secret tokens, resolves exact semver versions including prerelease/build metadata directly, resolves `*`, `x`, segment wildcards, caret ranges, tilde ranges, whitespace-AND comparator ranges, and `||` disjunction ranges from local/file/HTTP/HTTPS registry `index.json` into exact locked versions, preserves the original range as `requested_version`, and adds stable `fnv1a64` checksums. HTTP/HTTPS registry index requests use `Authorization: Bearer <token>` when `auth_token_env` is present. `--check` compares the existing lockfile without writing.

`orv fetch [dir-or-orv.toml] --out <dir>` verifies that `orv.lock` matches `orv.toml`, materializes path dependencies, local/file registry dependencies, and HTTP/HTTPS registry `/<package>/<version>/source-bundle.json` artifacts into source-bundle cache artifacts, sends `Authorization: Bearer <token>` for registry entries with `auth_token_env`, and writes `deps-manifest.json`.

`orv add <pkg> <version> [--dev] [--path <path>] [--registry <url>] [--manifest <dir-or-orv.toml>]` edits the selected dependency section and regenerates `orv.lock`; `orv remove <pkg> [--dev] [--manifest <dir-or-orv.toml>]` removes from the selected section and regenerates the lockfile.

`orv workspace new <member> [--root <dir>] [--name <name>]` creates a basic member project and records the relative member path in root `orv.toml` `[workspace].members` with resolver `2`.

`orv workspace graph [root] [--view] [--out <dir>]` reads root `[workspace].members`, loads each member entry through the shared ProjectGraph pipeline, records member graphs/files/dependencies, and emits path dependency edges between workspace members. Workspace path dependency edges include dependency package/section, target member name/version, requested version when present, and reject requested versions that do not match the target member version. With `--out`, it writes `workspace-graph.json`; with `--view`, it writes `workspace-graph.json` plus a static `index.html` workspace member/dependency graph view with member/edge search filtering.

`orv workspace lock [root] --out <dir>` reads the same workspace graph, orders members dependency-first from path dependency edges, writes per-member lockfiles under `members/<member>/orv.lock`, and emits `workspace-lock.json` with member project metadata, dependency lists, path dependency edges, lock order, and package counts without mutating member source directories.

`orv workspace fetch [root] --out <dir>` writes the same workspace graph and workspace lock artifacts, then materializes each member's lockfile into `members/<member>/deps/deps-manifest.json` and per-member dependency source-bundle caches. It emits `workspace-fetch.json` with dependency-first fetch order, member dependency manifests, and total package counts.

`orv workspace build [root] --out <dir> [--prod] [--incremental]` reuses the normal build pipeline for every workspace member, orders member builds dependency-first from path dependency edges, verifies each member build directory, writes member artifacts under `members/<member>`, and emits `workspace-build.json` plus `workspace-graph.json` as the top-level workspace build contract. With `--incremental`, unchanged member source-bundle input hashes are skipped when the previous verified build is still valid; rebuilt dependencies force dependent members to rebuild.

Exports also write a focused `runtime/panel.html` runtime companion, and trace-enabled exports write `trace/panel.html`. When `--build <dir>` is supplied, the export state, `native-host.json` `production` block, static shell, and `production/panel.html` companion include a Production panel summarizing checked client bundle targets with manifest capabilities plus preflight route-policy counts, DB, and commerce adapter contract artifacts. `native-host.json` is the native editor host manifest: it points at the shell/state/debug runner/result JSON/result HTML/runtime panel/production panel/trace panel artifacts, includes `panels[]` as an ordered native panel inventory, and carries runtime frame/panel contract, DAP adapter command, runner command, debug panel contract, result artifact panel contract, breakpoint/function-breakpoint/data-breakpoint/exception-filter/watch-expression argument contracts, trace stream runner contract, trace summary, per-frame trace source/production navigation, production client bundle summary/contracts, production preflight route-policy/adapter summary/contracts/panel contract, and trace/debug capabilities. `orv editor trace <dir> --trace <trace.json>` maps captured request frames back to source/production navigation and embeds the `/__orv/trace/events` open-ended EventSource trace transport URL when a server artifact has a stable listen endpoint. `orv editor trace-stream <dir> --events <trace-events.sse>` consumes a native-host EventSource body, extracts `orv:trace` snapshots plus `orv:trace.frame` deltas, and emits normalized editor trace stream JSON with latest trace payload and transport metadata.

Editor DAP control, breakpoint, function-breakpoint, data-breakpoint, exception-filter, and watch-expression commands are now first-class in the export contract: every exported control carries the exact `orv editor run-debug debug/session-runner.json --control <name>` runner command, every executable breakpoint row carries a `--breakpoint <path>:<line> --control continue` runner command plus `setBreakpoints` request preview, every ProjectGraph function/define row carries a `--function-breakpoint <name> --control continue` runner command plus `setFunctionBreakpoints` request preview, every runtime local row carries a `--data-breakpoint <local-name> --control continue` runner command plus `dataBreakpointInfo`/`setDataBreakpoints` request previews, every exception filter row carries an `--exception-filter <filter> --control continue` runner command plus `setExceptionBreakpoints` request preview, `native-host.json` mirrors control commands, breakpoint/function-breakpoint/data-breakpoint/exception-filter command inventory, breakpoint/function-breakpoint/data-breakpoint/exception-filter/watch-expression argument formats, a `debug.panel_contract` for host-side configuration/control/breakpoint/function-breakpoint/data-breakpoint/exception-filter/watch/result panels, the stable `debug/session-result.json` result artifact path with `panels.debug.session_summary`, `panels.debug.source_navigation`, `panels.debug.scopes`, locals, project variables, `panels.debug.function_breakpoints`, `panels.debug.data_breakpoints`, `panels.debug.exception_filters`, `panels.debug.watch_expressions`, and a `panel_contract` for native UI sections, and the rendered `debug/session-result.html` companion panel, and the static shell renders the selected runner command plus result artifact metadata next to the DAP request payload.

Trace transport is also mirrored for native hosts: when an editor export includes trace state and the build has a stable server listen endpoint, `native-host.json` includes `trace.transport` with the `/__orv/trace/events` EventSource URL plus `trace.stream_runner.command` for `orv editor trace-stream <build-dir> --events trace/events.sse`, `trace.status_filters`, and `trace.frames[*]` request summary/source/production navigation entries. Trace-enabled exports write `trace/panel.html`, mirror it as `artifacts.trace_panel_html`, `trace.panel_html_path`, and `trace.panel_artifact`, and include that artifact in the trace panel contract while the static shell renders the transport and stream runner panes.

- lifecycle: `initialize`, `shutdown`, notifications, unknown-method errors

- editor affordances: `textDocument/codeLens`, `textDocument/codeAction`, `workspace/executeCommand`, `textDocument/documentLink`, `textDocument/documentColor`, `textDocument/colorPresentation`, `textDocument/foldingRange`, `textDocument/selectionRange`, `textDocument/semanticTokens/full`, `textDocument/linkedEditingRange`, `textDocument/formatting`, `textDocument/rangeFormatting`, `textDocument/onTypeFormatting`
- navigation: `textDocument/definition`, `textDocument/declaration`, `textDocument/typeDefinition`, `textDocument/implementation`, `textDocument/moniker`
- hierarchy: `textDocument/prepareCallHierarchy`, `callHierarchy/incomingCalls`, `callHierarchy/outgoingCalls`, `textDocument/prepareTypeHierarchy`, `typeHierarchy/supertypes`, `typeHierarchy/subtypes`
- editing/introspection: `textDocument/references`, `textDocument/documentHighlight`, `textDocument/prepareRename`, `textDocument/rename`, `textDocument/hover`, `textDocument/signatureHelp`, `textDocument/inlayHint`, `textDocument/completion` with project symbols, language/directive/route-method snippets, `@body`/`@param`/`@query`/`@env` dot completions learned from open project source, `@param` names inferred from route `:param` path segments, and hover/reference/highlight support for `@body.*`/`@param.*`/`@query.*`/`@env.*` references

Code actions include diagnostic reveal commands plus edit quickfixes for incomplete `@route` heads, inserting default HTTP method/path text where the parser diagnostic lands.

Identifier edit/introspection surfaces (`references`, `documentHighlight`, `linkedEditingRange`, `prepareRename`, and `rename`) reject reserved language tokens and lowercase built-in `@` directives, so editor actions cannot rewrite or spotlight syntax keywords and core route/server directives as user symbols. Domain field names under `@body.*`/`@param.*`/`@query.*`/`@env.*` have dedicated reference/highlight lookup and stay outside rename/linked-editing because they are request-shape keys, not local identifiers.

The LSP session keeps `textDocument/didOpen`, full-sync `textDocument/didChange`, and `textDocument/didSave` text buffers so later file URI requests can run against current editor content, and clears that cache on save-without-text or `textDocument/didClose`.

## DAP

`orv dap serve --stdio` reuses the same project loader and ProjectGraph as CLI/LSP. It supports initialize/cancel/launch/attach/configurationDone, source/function/instruction/data breakpoints, breakpoint/goto/step-in target discovery, exception info, threads/stackTrace/scopes/variables, variable/expression mutation, evaluate/completions, loadedSources/modules/source, source-frame disassemble/readMemory, continue/reverseContinue/goto/stepIn/stepBack/restartFrame, disconnect/terminate/terminateThreads, and stdio lifecycle/output events. Snapshot-backed DAP Source objects include SHA256 source checksums for launch-time source integrity, `scopes`, `stepInTargets`, and `restartFrame` accept reported stack frame ids, current-frame scopes report `namedVariables`, `stackTrace` honors `startFrame`/`levels`, `variables` honors `start`/`count` plus `filter=named|indexed` for editor-side paging, and `setInstructionBreakpoints` verifies `orv:frame:N` pseudo-instruction references after launch so `continue` can stop on runtime frames even when a client drives the source-frame disassembly surface.

Runtime debug state comes from the reference runtime debug trace. Long-running `@server` launches start as a paused runtime frame and expose `continue`/`pause` event-loop control. `attachRuntime=true` or the standard `attach` request can attach a child `orv run <program>` process; `attachRuntimeMode="inProcess"` starts an attached server thread in the DAP process and preserves the same DB handle across top-level prefix statements, server boot body, and route handlers. In-process request frames are exposed through variables/evaluate/completions and can be flushed to `orv.production.trace` JSON via `runtimeRequestTracePath`, normal `orv run-artifact/run-build --trace <path>`, or `ORV_RUNTIME_REQUEST_TRACE_PATH`. When request trace capture is enabled, the reference HTTP runtime also serves an EventSource-compatible snapshot at `/__orv/trace/events`.

Static int `pow` operands outside the runtime-accepted exponent range `0..=63` stay on the generated native route path and return the same native arithmetic-failure class instead of forcing a reference bridge fallback.

Single `else if` response chains whose guard/body slices stay inside the same native-lowered request-body/route/query feature set are also direct-lowered; the generated handler emits ordered guarded returns and a final unconditional response.

The same chain shape is supported when the final unconditional `@respond` is a following route-handler statement instead of the last `else` branch.

`else { if ... }` block wrappers are normalized to the same ordered response chain when the block contains only that nested `if`.

Route path-param matching now recognizes static suffix segments such as `:userId.ics`; analyzer diagnostics reject `@param.<name>` references not declared by the current route path.

Request-body scaled captured arithmetic such as `{ quantity: (@body.quantity as int) + ((@body.bonus as int) * 2) }`, `{ quantity: ((@body.bonus as int) * 2) + (@body.quantity as int) }`, `{ quantity: (@body.quantity as int) - ((@body.bonus as int) * 2) }`, `{ quantity: ((@body.bonus as int) * 2) - (@body.quantity as int) }`, `{ cents: (@body.quantity as int) * ((@body.unit_price as int) * 100) }`, `{ unit: (@body.total as int) / ((@body.parts as int) * 100) }`, or `{ remainder: (@body.total as int) % ((@body.parts as int) * 10) }` is direct-lowered as `add_scaled`/`sub_scaled`/`rsub_scaled`/`mul_scaled`/`div_scaled`/`rdiv_scaled`/`rem_scaled`/`rrem_scaled` native metadata. Generated handlers parse both captured operands, checked-multiply the scaled operand, then checked-add/sub/mul/div/rem the result so these guarded commerce-style routes no longer require the reference bridge. Scaled captured comparisons in response fields and route guards, such as `{ available: (@body.quantity as int) <= ((@body.stock as int) * 10) }`, `{ covered: ((@body.minimum as int) * 100) <= (@body.total as int) }`, and `if (@body.quantity as int) <= ((@body.stock as int) * 10) { @respond ... }`, are also direct-lowered with checked scale multiplication before comparing.

Request-body captured product arithmetic/comparison such as `{ total: (@body.fee as int) + ((@body.quantity as int) * (@body.unit_price as int)) }`, `{ total: (@body.fee as int) - ((@body.quantity as int) * (@body.unit_price as int)) }`, `{ total: ((@body.quantity as int) * (@body.unit_price as int)) - (@body.fee as int) }`, `{ bundles: (@body.total as int) / ((@body.quantity as int) * (@body.unit_price as int)) }`, `{ remainder: ((@body.quantity as int) * (@body.unit_price as int)) % (@body.total as int) }`, `{ total: ((@body.quantity as int) * (@body.unit_price as int)) + 25 }`, `{ remaining: 1000 - ((@body.quantity as int) * (@body.unit_price as int)) }`, `{ covered: ((@body.quantity as int) * (@body.unit_price as int)) <= 1000 }`, `{ covered: (@body.total as int) <= ((@body.quantity as int) * (@body.unit_price as int)) }`, `if (@body.total as int) <= ((@body.quantity as int) * (@body.unit_price as int)) { @respond ... }`, or `if ((@body.quantity as int) * (@body.unit_price as int)) <= 1000 { @respond ... }` is direct-lowered as `add_product`/`sub_product`/`rsub_product`/`mul_product`/`div_product`/`rdiv_product`/`rem_product`/`rrem_product`, `*_product_static`, `*_product` comparison metadata, or condition secondary operand metadata. Generated handlers parse captured product operands, checked-multiply them, then checked arithmetic or compare against the base captured field or static operand so common cart-total and discount routes avoid the reference bridge.

Scaled-product response fields and guards such as `{ total: (@body.base as int) + (((@body.quantity as int) * (@body.unit_price as int)) * 100) }`, `{ total: (((@body.quantity as int) * (@body.unit_price as int)) * 100) + (@body.base as int) }`, `{ total: (@body.base as float) + (((@body.price as float) * (@body.quantity as float)) * 0.5) }`, `{ covered: (@body.total as int) <= (((@body.quantity as int) * (@body.unit_price as int)) * 100) }`, and `if (@body.total as int) <= (((@body.quantity as int) * (@body.unit_price as int)) * 100) { @respond ... }` are direct-lowered as `*_scaled_product` response metadata or condition `operand_scale_json` plus secondary operand metadata. The base captured field stays primary, `operand_json`/`operand_scale_json` carries the static scale, and `operand_*` plus `secondary_operand_*` carry the two product factors; generated handlers compute the product and scale with checked int arithmetic or finite float arithmetic before applying the response operation or branch.

Product-product response fields such as `{ total: ((@body.quantity as int) * (@body.unit_price as int)) + ((@body.fee_units as int) * (@body.fee_value as int)) }` and `{ covered: ((@body.quantity as int) * (@body.unit_price as int)) <= ((@body.stock as int) * (@body.reserve_price as int)) }` are direct-lowered as `*_product_product` metadata. The left product keeps the primary captured field plus `operand_*`, the right product uses `secondary_operand_*` and `tertiary_operand_*`, and generated handlers compute both products before checked arithmetic or comparison.

Captured triple-product response fields such as `{ total: (@body.base as int) + (((@body.quantity as int) * (@body.unit_price as int)) * (@body.bundle_count as int)) }`, `{ total: (((@body.quantity as int) * (@body.unit_price as int)) * (@body.bundle_count as int)) + (@body.base as int) }`, `{ total: (@body.base as float) + (((@body.price as float) * (@body.quantity as float)) * (@body.multiplier as float)) }`, `{ covered: (@body.total as int) <= (((@body.quantity as int) * (@body.unit_price as int)) * (@body.bundle_count as int)) }`, or the equivalent left-side comparison are direct-lowered as `*_triple_product` arithmetic/comparison metadata. The base captured field stays primary while the three product factors use `operand_*`, `secondary_operand_*`, and `tertiary_operand_*`; generated handlers parse all factors, validate the triple product, then apply checked int arithmetic/comparison or finite float arithmetic/comparison before writing JSON. Matching request-body guards such as `if (@body.total as int) <= (((@body.quantity as int) * (@body.unit_price as int)) * (@body.bundle_count as int)) { @respond ... }` use the same operand contract and stay on the direct native dispatch path.

Product-product guards such as `if ((@body.quantity as int) * (@body.unit_price as int)) <= ((@body.stock as int) * (@body.reserve_price as int)) { @respond ... }` use the same condition `operand_*`/`secondary_operand_*`/`tertiary_operand_*` contract and stay on the direct native dispatch path.

## DB Operations

Runtime `@db` currently uses an in-memory execution model with explicit JSON snapshot and WAL APIs: `@db.save/load`, `@db.wal(path)`, `@db.checkpoint()`, `@db.savepoint()`, and `@db.rollback(point)`. `@server` boot body DB setup is carried into route handlers, so a server-level SQLite handle such as `let shopdb = @db.connect(@env.SHOP_DATABASE_URL ?? "sqlite://data/shop.sqlite")` can be captured by routes and persist product/member/order/payment/shipment mutations across runs. The shop scaffold stores member signup rows, login session rows, cart item rows, and AuditEvent rows for product/member/session/cart/order/checkout/payment/shipping/webhook operations in the same SQLite handle, `GET /catalog` renders persisted Product rows for shoppers, `GET /cart` renders persisted CartItem rows, `GET /account/sessions` renders active Session rows for browser account inspection, `GET /admin/summary` uses the handle's `count` aggregation to expose product/member/order/payment/shipment/webhook/audit row counts for operators, and the admin read-model routes render persisted Product, Order, Payment, Shipment, WebhookEvent, and AuditEvent rows for browser inspection. `@db.connect` accepts the reference `memory://` adapter, a local WAL-backed `file://path` adapter, and a SQLite-backed `sqlite://path` adapter that stores ORV table metadata plus row JSON in a real SQLite file while preserving current query semantics. PostgreSQL/MySQL adapter URLs return external status handles until real provider implementations exist; those handles expose `adapterStatus`, `url`, provider, and `runtime.queryMethods` metadata, query/mutation methods fail explicitly, and build/prod deploy artifacts record those URLs as external `db_endpoints` without creating Compose volumes and emit a checked `deploy/db-adapters.json` operator contract. Build/deploy/server runtime artifacts include the `db_adapter` runtime feature when source uses `@db.connect`. Production deploy persistence scans `@db.wal("relative/path.jsonl")`, `@db.connect "file://relative/path.jsonl"`, and `@db.connect "sqlite://relative/path.sqlite"`, records SQLite files as `db_paths`, records env-configured DB adapters as `db_env`, and mounts eligible parent directories into Compose.

Runtime commerce adapters currently provide local reference handles: `@payment.connect("test://local").capture(...)` returns captured payment metadata and `@shipping.connect("test://local").book(...)` returns shipment booking metadata. The shop scaffold uses env-overridable local handles, `@payment.connect(@env.PAYMENT_ADAPTER_URL ?? "file://data/payments.jsonl")` and `@shipping.connect(@env.SHIPPING_ADAPTER_URL ?? "file://data/shipments.jsonl")`, so unset env defaults append and sync local JSONL records before matching DB rows are persisted, while deploy operators can point the same source at external HTTP adapters. `http://host/path` handles are reference external adapters: capture/book calls POST `{"kind","payload"}` JSON to the configured HTTP endpoint and return its JSON response as an ORV value. Prod deploy persistence records relative file adapter paths as `record_paths` and mounts their parent directories into Compose; HTTP adapter URLs are recorded as `commerce_endpoints` in deploy/container persistence and listed in the runbook without adding volumes. When the adapter URL comes from `@env.NAME ?? "default-url"`, deploy/container persistence also records `commerce_env` with the env name and default URL, the generated Compose file forwards that env with the default fallback, and the runbook lists the override knob. `deploy/commerce-adapters.json` mirrors those adapters into a checked operator contract for payment capture and shipping booking wiring. Builds that use these connections record `payment_adapter` and `shipping_adapter` runtime features in build, server runtime, and deploy artifacts. Provider-specific reference handles are available for `@payment.connect("stripe://...")` and `@shipping.connect("carrier://...")`; prod artifacts record them as `mode: "provider"` with `provider` names instead of pretending they are HTTP endpoints or local record logs, provider-mode artifacts expose required/optional credential env contracts for Stripe and carrier handles, runtime provider responses expose credential status metadata without leaking the configured values, and Stripe handles can verify reference webhook signatures with `payments.verifyWebhook({ payload, signature })`. The shop scaffold exposes that as `POST /webhooks/stripe`, persists idempotent `WebhookEvent` rows keyed by event id, and includes the persisted count in `GET /admin/summary`.

The reference HTTP server sets a built-in member session cookie for successful `POST /members/login` responses that return `session.id`: `orv_session=<id>; Path=/; Max-Age=86400; HttpOnly; SameSite=Lax; Secure`. If the response session carries `role`, it also emits `orv_session_role=<role>` with the same cookie defaults. String session ids and roles must already be cookie-safe ASCII scalars; unsafe values are not emitted. The reference `hash` namespace supports `hash.sha256`, Argon2 `hash.password`, and `hash.verify`; the shop scaffold stores only `passwordHash` values and rejects login when verification fails. Route handlers can use `@session.id`/`@session.role` to read those cookies, `@session required` to return `401 { err: "session_required" }`, and `@Auth required role="admin"` to return `401 { err: "auth_required" }` without a session or `403 { err: "role_required" }` for a mismatched role before later handler statements run. HTML GET responses receive a reference `orv_csrf=orv-reference-csrf` cookie, and route handlers can use `@csrf` to require a matching `x-csrf-token`, `x-orv-csrf-token`, `_csrf`, or `csrf` submitted token; mismatch returns `403 { err: "csrf_token_required" }` before later handler statements run. The shop product, member, login, order, cart, checkout, payment, and shipment browser mutation forms include the hidden `_csrf` token and their matching POST routes use `@csrf`; `POST /webhooks/stripe` remains signature-verified instead of CSRF-gated because it is a provider callback endpoint. The server also applies built-in one-minute route limits to the MVP shop mutation hotspots: `POST /members/login` and `POST /checkout` allow 10 requests per client IP, and `POST /webhooks/stripe` allows 60 requests per client IP. Explicit `@rateLimit` policies can override those limits, use request keys such as `@body.memberId`/`@query.memberId`/`@session.id`, or use `exempt` to remove the built-in limit for a route. Exceeded requests return `429 Too Many Requests` before handler execution.

The shop `POST /cart/items` route persists validated member/product cart rows for the browser `GET /cart` view, and `POST /checkout` composes member lookup, stock reservation, payment capture, shipment booking, and final order status update in one browser/API path while preserving the separate member/order/payment/shipment routes for explicit operations.

CLI DB commands provide schema/data dry-run, drift verification, apply/migrate with history, rollback, local backup/restore, hash-verified WAL recovery, archive manifest generation, `file://` archive target copies, HTTP `POST` archive target uploads plus HTTP `GET` restore fallback, S3-compatible `s3://` archive targets through `ORV_DB_ARCHIVE_S3_ENDPOINT` with optional bearer auth or `ORV_DB_ARCHIVE_S3_AUTH=aws-sigv4`, bounded retry for transient HTTP/S3-compatible upload/download failures via `ORV_DB_ARCHIVE_HTTP_RETRY_ATTEMPTS`, manifest-relative source WAL resolution, raw-WAL/archive point-in-time restore, crash/recovery matrix reporting via `orv db crash-matrix --out <report.json>`, and history squash.
